🔍 Audit 12 — Valeurs hardcodées non-adaptatives

📅 2026-05-15 🔢 SW trainhub-v239 🌿 claude/diagnose-program-display-U2WwA 📁 5 fichiers audités

js/engine.js — 6 violations mineures

⚠️
getLPIncrement() — ligne ~4266
if (gender === 'F' || bw < 65) return LP_CONFIG.increments.composé_leger;
Seuil absolu 65 kg pour séparer les incréments LP "léger" / "lourd". Un compétiteur de 64 kg reçoit le même incrément qu'un débutant de 55 kg. Non adaptatif au niveau ou à la force relative.
Remplacer par LP_INCREMENT_BY_LEVEL[level] ou ajouter un ratio DOTS-dépendant.
⚠️
shouldExitLP() — ligne ~1818
if (bw > 0 && (squat >= bw * 1.0 || bench >= bw * 0.8)) { return { exit: true, reason: 'performance' }; }
Ratios BW×1.0 (squat) et BW×0.8 (bench) pour sortir de LP. Non adaptés au genre : bench = 0.8×BW est élite pour une femme, mais intermédiaire pour un homme. Pas de différenciation par level ou gender.
Utiliser LP_EXIT_RATIOS[gender][lift] ou croiser avec les percentiles de STRENGTH_STANDARDS_FEMALE.
⚠️
analyzeAthleteProfile() — ligne ~2802
var dlSqRatio = raw.deadlift / raw.squat; if (dlSqRatio !== null && dlSqRatio < 1.10 && level !== 'debutant') {
Seuil 1.10 hardcodé directement dans la fonction. Incohérent : tous les autres ratios (squat_bench, row_bench…) passent par STRENGTH_RATIO_TARGETS. La table n'a pas d'entrée dl_sq.
Ajouter STRENGTH_RATIO_TARGETS.dl_sq = { ideal:[1.10,1.25], alert:1.10, danger:1.00 } et référencer depuis la fonction.
⚠️
getMRV() / getMEV() — lignes ~1184, ~1191
// getMRV return isFemale ? Math.round(base * 1.15) : base; // getMEV return isFemale ? Math.round(base * 1.10) : base;
Multiplicateurs féminins universels (×1.15 MRV, ×1.10 MEV) appliqués à tous les groupes musculaires sans distinction. Les fessiers/ischio nécessitent bien plus de volume pour les femmes que les triceps — un coefficient global efface cette nuance.
Ajouter une colonne female_mult par muscle dans MUSCLE_VOLUME_TARGETS avec des valeurs spécifiques (fessiers×1.30, pecs×1.10, triceps×1.05…).
⚠️
calcE1RMFrom5RepTest() — ligne ~3294
var e1rm = weight / (1.0278 - 0.0278 * reps); return Math.round(e1rm * 0.85 / 2.5) * 2.5;
Escompte 15% sur le e1RM estimé depuis un test 5RM. Aucun commentaire de source. Non adaptatif : un athlète élite a une courbe force-endurance plus plate (0.90 serait plus juste) qu'un débutant (0.80 acceptable).
Documenter la source (Helms 2017?) et/ou indexer par level : { debutant:0.80, intermediaire:0.85, avancé:0.88 }.
⚠️
calcStartWeightFromRPE5Test() — ligne ~4272
return Math.round(e1rm * 0.70 / 2.5) * 2.5;
Escompte 30% universel pour le poids de démarrage LP. Trop conservateur pour un intermédiaire qui reprend, trop généreux pour un vrai débutant avec technique précaire. Sans documentation.
Table LP_START_PCT[level] : débutant→0.65, intermédiaire→0.72, avancé→0.78.

js/app.js — 3 violations mineures

⚠️
wpGeneratePowerbuildingDay() — ligne 22139
if (phase === 'deload') { weight = wpRound25(weight * 0.80); setsCount = Math.ceil(setsCount / 2); rpe = 6; }
Deload uniforme : -20% charge, séries ÷2, RPE 6. Valeurs raisonnables (standard RP Strength) mais identiques que le deload soit préventif (SRS 65, forçé par planning) ou réactif (SRS 35, fatigue réelle). Un athlète léger en deload préventif pourrait maintenir -10% seulement.
Lire le SRS : deloadPct = srs < 40 ? 0.75 : 0.80, rpe = srs < 40 ? 5.5 : 6.
⚠️
wpApplyImbalanceCorrections() — lignes ~21164, ~21176, ~21195, ~21207
if (ratios.row_bench.value < 0.90) { /* Seal Row */ } if (ratios.bench_squat.value > 0.83) { /* Presse */ } if (ratios.squat_deadlift.value < 0.80) { /* Squat Pause */ } if (ratios.ohp_bench.value < 0.60) { /* OHP */ }
4 seuils hardcodés dans la logique de prescription, divergents de STRENGTH_RATIO_TARGETS (ex: row/bench alert = 0.85 dans la table vs 0.90 ici). Deux sources de vérité pour les mêmes ratios.
Lire les seuils depuis STRENGTH_RATIO_TARGETS[key].alert pour garantir la cohérence avec analyzeAthleteProfile().
⚠️
isFatigued() (via program.js) — ligne ~193 de program.js
var highVol = (totalVolume2Weeks / 2) > 20000; // 20 tonnes/semaine
Seuil tonnage absolu 20 000 g/semaine. Un athlète de 120 kg en phase force dépasse ce seuil normalement (5×5 @ 200 kg = 5 000 g × 3 lifts = 15 000 g + accessoires). Non calibré au BW ou au niveau.
var bwRef = (db.user && db.user.bw) || 80; var highVol = (totalVolume2Weeks / 2) > bwRef * 250;

js/coach.js — 1 violation mineure

⚠️
computeReadinessScore() — ligne ~481
raw = (acwrScore * 0.60) + (subjScore * 0.20) + (trendScore * 0.20);
Pondérations ACWR/Subjectif/Trend fixes, documentées dans CLAUDE.md mais pas dans le code source. Non adaptatives : si HRV est disponible, les poids changent (ACWR 40%, HRV 30%, Subj 15%, Trend 15%) mais les fallback sans HRV restent fixes indépendamment du niveau ou du profil.
Ajouter commentaire // Poids validés CLAUDE.md §4 SRS ; envisager une pondération adaptée au profil onboardingProfile (senior → Subjectif ×0.30, compétiteur → ACWR ×0.70).

js/program.js — 1 violation critique + 1 mineure

🔴
VOLUME_LANDMARKS (constante globale) — lignes 37–48
// program.js — VOLUME_LANDMARKS const VOLUME_LANDMARKS = { 'Dos': { mev: 10, mav: 20, mrv: 26 }, 'Pectoraux': { mev: 8, mav: 16, mrv: 22 }, ... }; // engine.js — MUSCLE_VOLUME_TARGETS (source officielle) var MUSCLE_VOLUME_TARGETS = { dos: { MEV: 10, MAV_low: 14, MAV_high: 22, MRV: 25 }, pecs: { MEV: 8, MAV_low: 12, MAV_high: 20, MRV: 22 }, ... };
Deux tables de volumes divergentes pour les mêmes muscles. Dos MRV : 26 (program.js) vs 25 (engine.js). Pas de MAV_low/MAV_high dans program.js. Les fonctions getMRV()/getMEV() d'engine.js et le calcul d'imbalance de program.js utilisent des références différentes — un utilisateur reçoit des recommandations contradictoires selon le chemin de code emprunté.
Supprimer VOLUME_LANDMARKS de program.js. Exposer MUSCLE_VOLUME_TARGETS depuis engine.js comme source unique. Adapter les références dans program.js.
⚠️
analyzeMuscleBalance() — appel isFatigued() — ligne ~193
var highVol = (totalVolume2Weeks / 2) > 20000;
Cf. app.js ci-dessus — même seuil absolu tonnage sans calibrage BW.

index.html — 1 violation mineure (admin)

⚠️
_tryActivateGodMode() — ligne 3476
const ADMIN_EMAIL = 'aurelien.cofypro@gmail.com';
Email personnel hardcodé pour l'accès God Mode. N'influence pas la prescription d'entraînement (portée : admin seulement). Risque opérationnel : si l'email change, le God Mode est bloqué sans modification du code. La vérification Supabase profiles.is_admin (ligne 3521) est suffisante à elle seule.
